# Legacy Known Bugs & Anti-Patterns

> A prioritized register of real defects, security holes, and design smells in the legacy Pawductivity (Flutter app + Go/Gin backend). Everything here is a **thing to consciously NOT repeat** in the local-first React Native / Expo rebuild. Sourced from the aggregated edge-cases finding, the auth deep-dive, and the completeness critic; every constant cited was re-verified against `old/`.

## How to read this file

- **Severity** — impact if carried into the rebuild: **Critical** (security/data-loss/store-rejection), **High** (user-visible wrong behavior or exploit), **Medium** (fragility / correctness-under-edge), **Low** (cosmetic / cleanup).
- **Rebuild action tag** — one of `[DROP]` (delete the concept), `[CHANGE]` (keep concept, new implementation), `[NEW]` (add missing safeguard), `[PRESERVE]` (rare — keep as-is), `[DECIDE]` (open product/eng decision).
- Legacy paths are relative to `old/`. Line numbers verified at time of writing.
- Many issues **evaporate for free** in the rebuild because there is no server, no JWT, no network, and a single local user. Those are marked "dissolved by local-first" so the team knows the win is structural, not something to re-engineer.

### Severity roll-up

| Severity | Count | Theme |
|---|---|---|
| Critical | 7 | Shipped secrets, auth bypass, unauthenticated money/coins endpoints, plaintext password on device, Play-policy foreground service |
| High | 9 | Reward/XP discrepancies, IDOR, free-coin & referral farming, wrong-row SQL bind, timer stops when backgrounded |
| Medium | 10 | Schema drift, missing permissions, no 401/refresh, non-atomic read-modify-write, recurrence/timezone fragility |
| Low | 6 | Dual duplicate stacks, dead stubs, cosmetic mis-bindings |

## 1. Security — Critical (all `[DROP]`; dissolved by local-first)

The entire remote auth/payment stack is deleted in the rebuild (no server, single local profile). These are documented so the team never reintroduces the pattern if an optional cloud layer is added later.

| # | Issue | Severity | Legacy location | Why it is bad | Rebuild action |
|---|---|---|---|---|---|
| 1.1 | **Hardcoded JWT signing secret `"secret"`** shipped in the binary | Critical | `Pawductivity_BE/internal/controllers/auth.controller.go:60,115`; `internal/middleware/jwtMiddleware.go:27,52` | The HS256 secret is the literal string `"secret"` in four places. Anyone can forge a valid token for **any** `id` claim — full account takeover of every user. | `[DROP]` No JWT at all. Identity = one local profile. If cloud sync is ever added, secrets live server-side only, never in the client. |
| 1.2 | **Hardcoded AES key/IV** for "password encryption" baked into the app | Critical | `Pawductivity_App/lib/features/user/presentation/pages/auth/login.dart:36-37`; `register.dart:45-46` | Key `'my 32 length key................'` (32 B) and IV `'1234567890123456'` (16 B) are hardcoded and mirrored in server env `AES_KEY`/`AES_VI`. Client-side "encryption" with a shipped key gives **zero** confidentiality (trivially extracted); the **static IV** makes ciphertext deterministic (same password → same blob). It is security theater over plain HTTPS. | `[DROP]` No client crypto. There is nothing to protect in transit (no server). Transport security, when any remote is added, is TLS + a server-verified token — never a shared symmetric key in the app. |
| 1.3 | **JWT is a fixed 31-day token, no refresh, no revocation** | Critical | `auth.controller.go:57,112` (`time.Now().Add(time.Hour * 744)`); `/token` only validates | Logout merely deletes the local copy; the token stays valid server-side for up to 31 days. A leaked token cannot be revoked. | `[DROP]` No tokens. See also §4.1 (no 401/refresh handling anywhere). |
| 1.4 | **Google Sign-In never verifies the Google token** — auth bypass | Critical | `auth.controller.go:94-126`; `internal/repository/user.repository.go:290-372` | `GoogleSignInRequest` binds only `{email, name}`; the access `token` field is **ignored**. Anyone can `POST /google-sign-in` with an arbitrary email+name and receive a valid 31-day JWT. New Google users are also created with `password=""` (empty), an unusable-login account with no set-password flow. | `[DROP]` for MVP. If social sign-in returns later it is only to key an **optional** cloud sync blob, and the ID token MUST be verified server-side. |
| 1.5 | **Plaintext user password persisted in secure storage** | Critical | `Pawductivity_App/lib/features/user/presentation/bloc/user/remote/remote_auth_bloc.dart:59` (`_storage.write(key: 'password', value: event.password)`) | During signup the **plaintext** password is written to `flutter_secure_storage` under key `'password'` and only cleared on the next `StoreCredential` (line 55) — so it lingers on-device after signup. `flutter_secure_storage` is OS-encrypted at rest but the app has no business retaining the raw password at all. | `[DROP]` No passwords exist in the rebuild. The **only** secret the new app holds is the Anthropic API key for client-side Claude calls — store it in `expo-secure-store` / MMKV, never a user password. |
| 1.6 | **`POST /purchase/coin` grants coins with zero payment verification** | Critical | `Pawductivity_BE/internal/repository/purchase.repository.go` (`buy_coins`); route live but no UI calls it | Any authenticated user can credit themselves **unlimited** coins. Signup itself abuses the same `buy_coins` proc to grant the 200 starting coins. A live authenticated free-money endpoint. | `[DROP]` No server, no purchasable coins. Coins are earned locally by task completion only; balance is a single local integer with a `>= 0` invariant enforced in a local transaction. |
| 1.7 | **`POST /premium/webhook` (Midtrans) is unauthenticated and skips signature verification** | Critical | `Pawductivity_BE` premium webhook handler; noted in Premium & Backend findings | Anyone can `POST {order_id, transaction_status:'settlement'}` to grant themselves premium. No `signature_key` check. A second real-money surface (Midtrans WebView Snap flow, `payment.dart` + `payment_web_view.dart`) also shipped. | `[DROP]` Midtrans entirely. Entitlement in the rebuild comes from `react-native-iap`/RevenueCat; **server-side receipt verification is the one thing that genuinely wants a backend** — document honestly. See `../migration/monetization-options.md`. `[DECIDE]` |

## 2. Economy & gamification correctness — the numbers do not agree

These are the **most user-visible legacy bugs** and the ones the rebuild must fix deliberately, because the gamification core (coins/XP/level) is being carried forward as a **local TypeScript service**.

| # | Issue | Severity | Legacy location | Why it is bad | Rebuild action |
|---|---|---|---|---|---|
| 2.1 | **Task-completion reward: three different numbers** | High | Grant: `task.repository.go:470` `CALL buy_coins(userId, estimatedTime/60)`. Display: `task.repository.go:234` `FLOOR(estimatedTime / 60 / 3) AS rewardCoins`. UI default: `task_template_model.dart:32` `reward: json['reward'] ?? 100` | A user is **granted `estimatedTime/60`** coins (≈1 per estimated minute) but the API **displays `FLOOR(estimatedTime/60/3)`** (≈⅓ of that). Worse, the API returns the key `rewardCoins` while Flutter reads `json['reward']` — which is absent — so the UI **always shows the fallback `100`**. Net: user sees **+100**, the "reward" field claims **≈est/60/3**, and the wallet actually gains **est/60**. Three disjoint numbers for one event. | `[CHANGE]` Pick ONE formula, compute it once locally, and show exactly what is granted. Recommended: grant = `Math.floor(estimatedSeconds/60)` coins, displayed identically. `[DECIDE]` confirm the intended coins-per-minute rate and whether it should be the /3 value. |
| 2.2 | **`needed_xp` seed 150 disagrees with the level-1 formula (160)** | High | Seed default `Pawductivity_BE/database/migration/model/user.model.go:9` `NeededXP int32 gorm:"default:150"`. Formula `task.repository.go:453` `neededXP = 10*level*level + 50*level + 100` → L1 = **160** | A brand-new user needs **150** XP for level 1→2, but every subsequent threshold uses the formula (160, 240, 340…). The first level-up is off by 10 and inconsistent with the curve. | `[CHANGE]` Seed `needed_xp` from the formula: `10*1²+50*1+100 = 160`. One source of truth for the curve. See `../../.claude/skills/gamification-xp-levels/SKILL.md`. |
| 2.3 | **`needed_xp` recomputed with stale pre-increment level inside the loop** | Medium | `task.repository.go:451-461` | The loop subtracts `neededXP`, then sets `neededXP = f(level)` using the **old** level, then increments `level` — so the next iteration's threshold lags one level. Combined with 2.2 the curve is doubly suspect. | `[CHANGE]` Reimplement cleanly: `while (xp >= needed) { xp -= needed; level++; needed = 10*level*level + 50*level + 100; }` — increment level **before** recomputing. |
| 2.4 | **Reward field never round-trips** | Medium | `TaskRequest` (POST) has no `reward` field; `GetTask` returns `rewardCoins`; Flutter reads `json['reward']` → default 100 | Reward is neither sent nor read correctly end-to-end; it is decorative. | `[CHANGE]` Compute reward at completion time from the task's estimate; never persist a stale display value. |
| 2.5 | **Starting-grant coins are an accidental side effect of the Cat's price** | Medium | `user.repository.go:122-180` — signup reads `animal id=2` (Cat) price and `CALL buy_coins(userId, price)`; Cat price = 200 | New users start with 200 coins **because** that is the Cat's price and animal id is hardcoded to 2. Change the seed catalog order/price and starting coins silently change. | `[CHANGE]` Make the first-run grant a **named constant** (e.g. `STARTING_COINS = 200`), decoupled from any catalog price. See `../data-model/seed-catalogs.md`. `[DECIDE]` confirm the intended amount. |
| 2.6 | **Referral coin reward silently lost on error** | High | `referral.repository.go` `UseCode` — two coin `UPDATE`s assign to `err` but it is never checked; line 56 overwrites the owner-update error with the redeemer-update error; function `return nil` unconditionally, deferred commit still runs | A failed coin `UPDATE` reports success while the redemption row commits — rewards silently not granted, and the first error is lost entirely. | `[CHANGE]`/`[DROP]` If referral survives, every mutation is checked and wrapped in one local transaction that rolls back atomically. |
| 2.7 | **Ledger sign ambiguity** — spends and credits both stored as positive `price` | Medium | `purchases` table: food/pet/clothes deduct but store positive `price`; `type='coins'` credits also positive | A naïve `SUM(price)` misrepresents net spend; every analytics query must special-case `type='coins'` as a credit. | `[CHANGE]` Local ledger uses **signed** deltas (or an explicit `direction` column) so aggregates are correct without special cases. |

## 3. Native / platform config — Play-policy & Android-13+ hazards

The rebuild's timer + reminder notifications must get the native surface right. Expo abstracts most of this, but the legacy manifest documents exactly what NOT to declare.

| # | Issue | Severity | Legacy location | Why it is bad | Rebuild action |
|---|---|---|---|---|---|
| 3.1 | **Foreground service typed `"location"` on a focus timer** | Critical | `android/app/src/main/AndroidManifest.xml` — `<service android:name="id.flutter.flutter_background_service.BackgroundService" android:foregroundServiceType="location" .../>` | A productivity timer declared as a **location** foreground service. On Android 14+ this is a policy violation (the app must actually use location) → **Play Store rejection** or runtime kill. | `[CHANGE]` In Expo, use the correct FGS type. A live countdown is best modeled as an **ongoing notification** (`expo-notifications`) + timestamp math, not a mistyped FGS. See `../../.claude/skills/focus-timer-and-background/SKILL.md`. |
| 3.2 | **Ghost `flutter_foreground_task` service with placeholder value** | Medium | Same manifest — `<service android:name="com.pravera.flutter_foreground_task.service.ForegroundService" android:foregroundServiceType="specialUse">` with `<property ... android:value="Whatever idk"/>` | A second FGS declaration for a plugin **not even in `pubspec.yaml`** — stale/broken, and `specialUse` with a junk subtype string is another review-rejection risk. | `[DROP]` No orphan native declarations. Only declare services actually used, with a real justification string. |
| 3.3 | **Missing runtime permissions** | Medium | Manifest declares only `INTERNET` and `FOREGROUND_SERVICE`. `home_screen.dart:82` calls `Permission.notification.request()` at runtime | Missing **`POST_NOTIFICATIONS`** → notification permission silently fails on Android 13+. Missing **`SCHEDULE_EXACT_ALARM`/`USE_EXACT_ALARM`** → no exact reminder/completion alarms. Missing **`RECEIVE_BOOT_COMPLETED`** → no reboot recovery/rescheduling. Missing `WAKE_LOCK`. No deep-link/App-Link intent filters (blocks referral deep-links). | `[NEW]` Declare all needed permissions via Expo config plugins and **prime them in onboarding**. `expo-notifications` for POST_NOTIFICATIONS; `expo-task-manager`/`expo-background-task` for periodic wake; reschedule on boot. See `../../.claude/skills/notifications-and-permissions/SKILL.md`. |
| 3.4 | **No reboot recovery for a running timer** | Medium | `flutter_background_service autoStart=false`, no boot receiver | A device reboot loses a running focus session (only reconstructable if the page is reopened and `start_time` is read). | `[CHANGE]` Persist `start_time` in MMKV; on app open/boot recompute elapsed from the timestamp. Never rely on a live tick surviving process death. |

## 4. Architecture & data-layer anti-patterns

| # | Issue | Severity | Legacy location | Why it is bad | Rebuild action |
|---|---|---|---|---|---|
| 4.1 | **No central auth interceptor, no 401/refresh handling** | Medium | `injection_container.dart` registers a single bare `Dio()` with no base-URL override and no interceptor; JWT attached per-endpoint via explicit `@Header('Authorization')` read from secure storage at each call site (`constant.dart` `SERVER_URI = https://fcfcvrer.pawductivity.id`) | Every call site re-reads the token; there is **no interceptor, no refresh, no 401 handling**. An expired/invalid token surfaces as ad-hoc per-bloc errors (often an infinite shimmer). The auth "gate" is literally "can I fetch `/api/user` with the stored JWT" (`main.dart` splash → `AppNavbar` on success, `RegisterPage` on error). | `[DROP]` The whole retrofit/Dio network layer is deleted — the rebuild has no network. Data access = synchronous local reads (expo-sqlite / Zustand+MMKV), offline by default. See `../../.claude/skills/local-first-data-layer/SKILL.md`. |
| 4.2 | **Dual old/new stacks wired simultaneously** | Low | `injection_container.dart` registers **both** `TaskApiService` + `TaskApiServiceOld`, `TaskRepository` + `TaskRepositoryOld`, `RemoteTaskBloc` + `RemoteTaskBlocOld`, plus `old_usecases` | Two parallel task stacks live at once (mid-migration debt). Same pattern in the timer: two `onStart` background-service entry points share notification id `888` (`background/background_service.dart` and inline in `task_management_screen.dart`), risking conflicting service definitions. And three timer screens (`task_screen_old` / `task_management_screen` / `task_timer_page`). | `[CHANGE]` Rebuild ships **one** implementation per concern. No parallel legacy shims. Reconcile the confirmed duplicate pairs (see §5.5) before writing code. |
| 4.3 | **Duplicate navigation systems & divergent screens** | Low | `config/routes/routes.dart` named routes **and** ad-hoc `Navigator.push(MaterialPageRoute)` scattered in pages; `calendar_page.dart` (navbar tab) vs `calendar_screen.dart` (`/calendar` route); `profile.dart` vs `profile_old.dart` | Two routing mechanisms and multiple divergent copies of the same screen — easy to edit the wrong one. | `[CHANGE]` One coherent navigator (expo-router / react-navigation). See `../../.claude/skills/navigation-and-app-shell/SKILL.md` and `navigation-map.md`. |
| 4.4 | **"Local persistence" is really `flutter_secure_storage` K/V, not the Floor DB** | Medium | Floor `UserDao`/`clothes`/`food` local entities are **dead code** (TODO comments admit purchase/feed never write locally); real client state is ~30 files hitting `flutter_secure_storage` keys: `auth_token`, `email`, `password`, `name`, `current_task_id`, `start_time`, `locked_pet_id`, `is_running`, `remaining_time`, `task_name` | The offline story is broken: inventory/food/clothes correctness depends **entirely on re-fetching** from the server, so the legacy app is effectively non-functional offline. The "local DB" was never wired. | `[CHANGE]` This is the whole point of the rebuild: real local-first with expo-sqlite (relational) + MMKV/Zustand (K/V + reactive state). Every mutation commits locally first. See `../data-model/sqlite-schema.md` and `../data-model/state-and-mmkv.md`. |

## 5. Data / schema drift, SQL bugs, and fragile derivations

| # | Issue | Severity | Legacy location | Why it is bad | Rebuild action |
|---|---|---|---|---|---|
| 5.1 | **Schema drift: `current_xp`/`needed_xp`/`profile_index` exist only in GORM migration** | Medium | Raw `Pawductivity_BE/database/script/pawductivity.sql` `users` table omits these columns; Go code `SELECT`s them; GORM `user.model.go:8-9` defines them | The app depends entirely on `AutoMigrate` having added the columns; on a raw-SQL-only DB, `GetUserLevel`/`GetUserById` would 500. Two divergent schema sources. | `[CHANGE]` Define the local `users` table **once** with exactly the columns used: `id, name, coins DEFAULT 0, level DEFAULT 1, current_xp DEFAULT 0, needed_xp DEFAULT 160, profile_index`. No AutoMigrate/raw-SQL split. |
| 5.2 | **Change-clothes UPDATE binds the wrong column → wrong/no row updated** | High | `Pawductivity_BE/internal/repository/clothing.repository.go:163` — `UPDATE petClothes SET wardrobeId=$1 WHERE userId=$2 AND petId=$3` bound with `(wardrobeId, userId, req.ClothesId)`; should be `req.PetId` | The `petId` placeholder receives the **clothes id**, so switching an already-dressed pet from one outfit to another matches on `petId == clothesId` and almost always updates the wrong row or none. Changing clothes on a dressed pet is **broken**. | `[CHANGE]` Rebuild the equip logic locally keyed by the real pet id; keep the equip/toggle semantics, drop the `userId` dimension (single user). See `../../.claude/skills/clothes-and-wardrobe/SKILL.md`. |
| 5.3 | **Non-atomic read-modify-write on health/coins (TOCTOU races)** | Medium | `FeedPet` `SET health = CAST(pet_health) + CAST(food_stats)` using a value read earlier; balance read early then deducted in a separate `UPDATE` | Concurrent feeds/purchases race; balance check can be passed by two purchases at once, mitigated only by an opaque `CHECK(coins>=0)` 500. | `[CHANGE]` Local single-user + a single SQLite transaction per mutation removes the race entirely — but still write it as one atomic statement, not read-then-write. |
| 5.4 | **`nil` transaction rollback panics on `Begin()` error** | Medium | `decreasePetHealth.routine.go:31-33`, `checkMembership.routine.go:31-33`, `purchase.repository.go` PurchaseWardrobe | `defer tx.Rollback()` runs on a `nil` tx when `db.Begin()` failed → panic/no-op. Also `PurchaseFood` overwrites the `playerFood` INSERT error with the next call's error, silently committing a coin deduction without adding the food. | `[DROP]` (server routines gone) / `[CHANGE]` local transactions check every step and roll back atomically. |
| 5.5 | **Server cron routines have no catch-up & share one timezone** | Medium | `decreasePetHealth.routine.go`, `checkMembership.routine.go` — single goroutine forever-loops keyed to **server-local** midnight | A server outage spanning midnight silently skips that day's health decay; all users share one server timezone. | `[CHANGE]` No daemon. Compute elapsed days from timestamps **on app open/resume** in the user's local time: `health -= min(daysElapsed, health)` floored at 0. See `../../.claude/skills/pet-companion-system/SKILL.md`. |
| 5.6 | **Timer trusts device wall-clock; drifts across backgrounding** | High | `countdown_manager.dart:40,82` (`start_time` ISO write, `DateTime.now().difference(startTime)`); background start/stop **commented out** at `countdown_manager.dart:53,58-62` | Elapsed is derived from wall-clock, so a clock change/DST can jump a task to complete or add a huge increment (an old comment even admits the backend must guard against "increment ketinggian" — too-high increment — but no client clamp exists). And because `BackgroundServiceHandler.start/stopForegroundService()` are **commented out**, backgrounding/killing the app **stops the visible countdown** entirely — only `start_time` survives. | `[CHANGE]` Persist `start_time`/`end_time` timestamps; compute remaining from them and **clamp** to `[0, allocatedTime]`; drive a live notification. Never trust accumulated ticks across backgrounding. See `../../.claude/skills/focus-timer-and-background/SKILL.md`. |
| 5.7 | **DB `CHECK(estimatedTime > 600)` silently rejects short tasks** | Medium | Task schema `CHECK` + `add_task_form` only validates `> 0` | Any task ≤ 10 minutes is accepted by the form but rejected server-side with an opaque 500; the edit form also drops the seconds component (`h*3600+m*60`), which can push a task below the check. | `[CHANGE]`/`[DECIDE]` Decide the real minimum focus duration and validate it in the UI; don't hide the rule in a DB constraint. |
| 5.8 | **Species/outfit derivation via fragile string interpolation & sentinels** | Medium | Client parses current clothes id out of the filename via `split('_')`/`replaceAll('.json','')` (`equip_clothes_listener.dart:78-88`); `animalId → type` silently defaults unknown ids to `'dog'`; `clothesId <= 0` (incl. the `-1` remove sentinel) collapses to default Lottie | Rendering depends on exact filename formatting and an overloaded `-1` sentinel ("remove" vs "no clothes"); an unknown species renders dog art; a missing asset throws at runtime. | `[CHANGE]` Store the equipped cosmetic as a real id/foreign key, resolve assets from a typed catalog map, and handle "no clothes" as an explicit null — no string parsing, no magic sentinels. See `../../.claude/skills/lottie-animation-engine/SKILL.md`. |
| 5.9 | **Recurrence & timezone fragility in reminders** | Medium | Monthly recurrence uses day-of-month equality (days 29/30/31 skip short months, no clamping); month view expands recurrence but the day/todo view matches exact Y/M/D so recurring reminders only appear on their original date; Postgres `DOW` (Sun=0) vs Flutter `weekday%7` index mapping | Recurring reminders display inconsistently across views; the DOW index mapping is easy to invert when reimplementing. | `[CHANGE]` One recurrence engine, clamp end-of-month, and a single tested weekday convention. See `../../.claude/skills/reminders-and-calendar/SKILL.md`. |
| 5.10 | **`INNER JOIN membership` hides users; missing membership → unreadable account** | Low | `GetUsers`/`GetUserById` inner-join membership | If membership creation fails mid-signup the account becomes invisible/errors. | `[DROP]` No membership table split for a single local user; entitlement is a local flag. See `../../.claude/skills/premium-and-monetization/SKILL.md`. |

| 6.6 | **Optimistic UI never rolled back on failure** | Medium | Food/Wardrobe purchase dialogs call the event then immediately `Navigator.pop()` regardless of success (no `insufficient`/`balance` check anywhere in `lib`); `PetList.updatePetHealth` animates the health bar up on tap and is never reverted if `/feed` fails; feeding at full health still DELETEs a food for `food_stats=0` benefit | Displayed state diverges from truth until the next re-fetch; a food can be wasted at 100 health. | `[CHANGE]` Update state **only after** the local transaction commits; block feeding when `health == 100`; surface failures inline (no silent pop). |

## 7. Meta-lessons for the rebuild (the patterns behind the bugs)

1. **One source of truth per number.** The reward (§2.1), XP curve (§2.2–2.3), and schema (§5.1) each had 2–3 disagreeing definitions. In the rebuild, define every constant and formula **once** (see `../data-model/seed-catalogs.md`, `../../.claude/skills/gamification-xp-levels/SKILL.md`) and derive everything from it.
2. **Commit-then-render, never render-then-hope.** Every legacy optimistic update (§6.6, §5.6) diverged from truth because it wasn't reconciled. Local-first makes the transaction synchronous — update UI from committed state only.
3. **Never trust the wall clock across process death.** Health decay (§5.5) and the timer (§5.6) must recompute from persisted timestamps on open/resume, clamped to valid ranges.
4. **No secrets in the client, ever.** JWT secret (§1.1), AES key (§1.2), and the plaintext password (§1.5) were all in the app. The rebuild holds exactly one secret — the Anthropic API key — and treats even that carefully.
5. **Declare only native config you actually use, correctly typed.** The mistyped/ghost foreground services (§3.1–3.2) are direct Play-rejection risks; the missing permissions (§3.3) fail silently on modern Android.
6. **Delete duplicates before writing new code.** Reconcile the confirmed mid-migration pairs — old/new task stacks, two timer service entry points, `calendar_page`/`calendar_screen`, `profile`/`profile_old` (§4.2–4.3) — so the rebuild starts from one clear model.
